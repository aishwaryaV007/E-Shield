# ExamShield Context & Scope
> Problem, product goals, boundaries, glossary, and status for the AI answer-sheet evaluator.

*Design / Planned — Not yet implemented*

---

## 1. Problem Statement

Universities evaluate lakhs of handwritten answer scripts each semester, managed by the
**Controller of Examinations (CoE)** and executed by evaluators under heavy workload. Manual
evaluation is **slow, inconsistent, and subjective**: different evaluators award different marks
for equivalent answers, standards drift across a batch, and nothing captures *how teachers grade*
so it can be reused. Revaluation disputes and backlogs follow.

---

## 2. Product Goals

- **Learn marking behaviour:** train a model on previously corrected scripts + teacher-awarded
  marks so grading follows the teachers' own standard, not hand-written rules.
- **Grade consistently:** every answer scored against the same learned standard, with measurable
  agreement (±1 mark) against teacher marks.
- **Explain every mark:** feedback + deduction reasons per answer.
- **Stay compliant & private:** the mark comes from the trained model, never an LLM; everything
  runs locally so scripts never leave the machine.

---

## 3. Scope & Boundaries

### In Scope
- **Phase 1 — training:** build a labeled dataset from the historical corpus; engineer features
  (semantic similarity, concept coverage, …); train + evaluate an XGBoost mark-predictor.
- **Phase 2 — evaluation:** handwriting OCR, question segmentation + answer-key matching, semantic
  scoring via the trained model, feedback, and an evaluated sheet (question-wise marks, total, %).
- Dashboard: training metrics + evaluated sheets.

### Out of Scope (Non-Goals)
- **LLM-generated marks:** prompting GPT/Claude/etc. to assign or predict marks is prohibited
  (Track 02). An LLM may only phrase feedback text.
- **Uploading scripts to cloud services:** privacy + offline constraints.
- **Online proctoring / browser lockdown:** not in the MVP.

---

## 4. Glossary

| Term | Definition |
| :--- | :--- |
| **Answer Script** | The physical handwritten booklet of a student's exam responses. |
| **Mark-predictor** | The trained regressor (XGBoost) that maps answer features → a mark. |
| **Answer Key** | The official model answer + rubric key-points + max marks per question. |
| **Semantic Similarity** | Meaning-based closeness of a student answer to the key (embedding cosine), not keyword overlap. |
| **Concept Coverage** | How many rubric key-points an answer covers / misses / contradicts. |
| **Percentage-based Marking** | Marks proportional to match quality (e.g. 90–100% → full marks). |
| **±1-mark accuracy** | Share of answers where the predicted mark is within 1 of the teacher's — the headline metric. |
| **Low-confidence answer** | An answer the OCR couldn't read reliably — flagged for human verification, never guessed. |
